# HomePage 날짜 상태 관리 의사결정: Zustand 대신 React Context 사용

## 문서 목적
HomePage에서 `selectedDate` 상태를 왜 Zustand가 아닌 React Context로 관리하는지 의사결정 근거를 기록한다.

## 대상 범위
- 상태: `selectedDate`, `setSelectedDate`
- 적용 범위: `HomePage` 하위 섹션
  - 월간 리포트
  - 루틴 리포트
  - 할 일 리포트

현재 구현은 `DateProvider` + `DateContext` + `useSelectedDate` 조합으로 구성되어 있다.

## 고려한 3가지 방법

### 1) React Context (채택)
- 방식: `HomePage` 루트에서 Provider로 감싸고 하위에서 `useContext`로 조회
- 장점
  - 상태 스코프를 HomePage로 한정 가능
  - 의존성 추가 없음
  - 구현/이해/유지보수 비용이 낮음
- 단점
  - 구독 단위 최적화가 제한적 (값 변경 시 소비자 리렌더 가능)

### 2) Zustand Store
- 방식: 전역 store 생성 후 필요한 컴포넌트에서 selector로 구독
- 장점
  - selector 기반 구독 최적화 용이
  - 상태/액션이 커질 때 구조화에 유리
- 단점
  - 현재 문제 크기 대비 과한 추상화
  - 전역 store 확장으로 책임 경계가 흐려질 수 있음

### 3) Prop Drilling
- 방식: 상위에서 하위까지 props로 전달
- 장점
  - 러닝커브/도구 의존성 없음
- 단점
  - 중간 컴포넌트가 상태 전달만 담당하게 되어 가독성/변경 용이성 저하
  - 트리 구조 변경 시 전달 경로 수정 비용 증가

## 채택 결론
HomePage 날짜 상태는 **로컬 페이지 스코프 UI 상태**이며, 현재 요구사항은 **읽기 + 단순 업데이트(set)** 중심이다.
이 조건에서는 React Context가 다음 이유로 가장 적합하다.

1. 상태의 책임 범위를 HomePage 내부로 명확히 제한할 수 있다.
2. 외부 상태관리 라이브러리 도입 없이 구현이 단순하다.
3. 현재 기능 복잡도(비동기 액션/영속화/미들웨어 부재)에 비해 충분한 해법이다.
4. 팀원이 코드를 읽을 때 의도가 직관적이다.

## 왜 지금은 Zustand가 아닌가
Zustand는 전역 공유, 복잡한 액션 흐름, 선택적 구독 최적화가 중요한 상황에서 강점이 크다.
그러나 현 시점의 `selectedDate`는 다음 특성을 가진다.

- 사용 범위가 HomePage로 제한됨
- 액션이 단순함(`setSelectedDate`)
- 전역 동기화/영속화 요구 없음

따라서 현재는 Zustand 도입 이득보다 구조 복잡도 증가 비용이 더 크다고 판단했다.

## 재검토 트리거 (Zustand 전환 기준)
아래 조건 중 2개 이상 충족 시 Zustand 전환을 재검토한다.

1. 날짜 상태를 여러 페이지/도메인에서 공통 사용
2. 날짜 상태 기반 파생 상태/액션이 증가
3. 컨텍스트 리렌더 비용이 성능 이슈로 확인됨
4. 영속화(예: 사용자별 마지막 선택 날짜 복원) 요구가 생김
5. 날짜 상태와 다른 도메인 상태 간 트랜잭션성 업데이트 필요

## 개선안 점검 (3가지)

1. Context 분리
- 개선 내용: 읽기 전용 값/업데이트 함수를 분리한 이중 컨텍스트 구조 검토
- 기대 효과: 불필요 리렌더 완화

2. Provider 경계 축소
- 개선 내용: 실제로 날짜 상태가 필요한 섹션만 Provider 하위로 감싸기
- 기대 효과: 컨텍스트 영향 범위 최소화

3. 리렌더 계측 도입
- 개선 내용: React DevTools Profiler 기반 월/주/일 이동 시 리렌더 측정 기준 수립
- 기대 효과: 전환 필요성(Context 유지 vs Zustand 전환)을 데이터로 판단

## 적용한 개선 선택
현재 단계에서는 **3번(리렌더 계측 기준 수립)**을 우선 적용 대상으로 선정한다.
이유는 성능 이슈 확인 전 구조 변경(1, 2)을 선행하면 불필요한 복잡도만 증가할 수 있기 때문이다.

운영 원칙:
- 먼저 측정하고,
- 병목이 확인되면 Context 분리 또는 Zustand 전환을 단계적으로 진행한다.
