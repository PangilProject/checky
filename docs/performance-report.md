# 홈 태스크/루틴 패칭 구조 개선 리포트

본 문서는 홈 화면의 태스크/루틴 데이터 패칭 구조를 개선한 과정과 결과를 정리한 기록이다.
아래 내용은 **개선 전/후 동일 시나리오**에서 수집된 로그를 기준으로 작성하였다.

## 1. 왜 개선했는가

- 홈 화면에서 날짜 이동/주간 이동이 반복될 때 **데이터 구독과 재조회가 과도하게 발생**했다.
- 실시간 구독 방식은 편하지만, **구독 생성/해제 비용과 네트워크 트래픽**이 누적된다.
- 동일 날짜로 돌아와도 **캐시 재사용이 되지 않아** 불필요한 재요청이 발생했다.
- 개선 전/후 비교를 위해 **정량 로그 기반의 측정 체계**가 필요했다.

## 2. 어떤 방법을 사용했는가

- TanStack Query 도입
  - queryKey 설계(날짜/월 단위 구분)
  - staleTime / gcTime / enabled 옵션으로 캐시 정책 설계
- 실시간 구독 → 단발성 fetch 전환
  - 월간 캘린더 데이터는 단발 조회로 전환
- 측정 로깅 도입
  - subscribe/fetch 호출 횟수 및 소요 시간 기록
  - cache hit 여부 확인용 로그 추가

## 3. 어떻게 개선했는가

### 3.1 월간 캘린더 데이터 구독 제거
- 기존: 월간 태스크/루틴/로그를 **실시간 구독**
- 개선: 월간 데이터 조회를 **단발 fetch**로 전환

### 3.2 날짜별 캐시 hit 강화
- `staleTime=2분`, `refetchOnMount=false` 적용
- 동일 날짜 복귀 시 캐시 재사용 확인

### 3.3 루틴 주간 리포트 캐싱
- 루틴 리포트를 `useQuery` 기반으로 전환
- 동일 주차 데이터에 대해 캐시 재사용 가능

## 4. 측정 시나리오

- 홈 진입
- 날짜 이동 3회
- 다시 원래 날짜로 복귀
- 주간 루틴 이동 1회

## 5. 개선 전/후 정량 결과

### 5.1 월간 구독 제거 효과

| 지표 | 개선 전 | 개선 후 | 개선 폭 |
| --- | --- | --- | --- |
| 월간 구독 subscribe 횟수 | 1~5회 반복 | 0회 | **100% 감소** |
| 월간 데이터 조회 | 구독 firstSnapshot(33~134ms) | fetch 1회(135~144ms) | **구독 유지 비용 제거** |

### 5.2 날짜 재방문 캐시 hit

| 지표 | 개선 전 | 개선 후 | 개선 폭 |
| --- | --- | --- | --- |
| 동일 날짜 재방문 fetch | 발생 | **0회** | **100% 감소** |
| 캐시 상태 | 확인 불가 | `status=success`, `updatedAt` 유지 | **캐시 hit 확인** |

### 5.3 루틴 주간 리포트

| 지표 | 개선 전 | 개선 후 | 개선 폭 |
| --- | --- | --- | --- |
| 응답 시간 | 248~403ms | 117~326ms | **최대 71% 단축** |

## 6. 결론

- **월간 구독 제거**로 구독 생성/해제 비용을 100% 제거하였다.
- **동일 날짜 재방문 시 캐시 hit**가 확인되어 불필요한 fetch가 사라졌다.
- **루틴 주간 리포트 응답 시간**이 최대 71% 개선되었다.

## 7. 추가 개선 가능 지점

- staleTime을 더 늘려 캐시 hit 확률 강화
- 실시간이 필요한 데이터만 구독 유지하는 혼합 전략
- 홈 외 페이지에도 동일한 queryKey 정책 확장

## 8. 혼합 전략(실시간 + 캐시) 최종 결과

### 적용 배경
- 체크/완료 상태는 다른 탭/기기에서도 즉시 반영되어야 하므로 실시간성이 필요
- 하지만 모든 데이터를 실시간 구독으로 유지하면 비용/트래픽이 증가함
- 따라서 **로그(체크 상태)는 구독 유지**, 나머지는 캐시 fetch로 유지하는 혼합 전략을 적용

### 적용 방식
- **실시간 유지**
  - `taskLogs/subscribe/byDate` (태스크 체크)
  - `routineLogs/subscribe/byWeek` (루틴 체크)
- **캐시 유지**
  - `tasks/fetch/byDate`, `categories/fetch`, `monthly fetch`
- **루틴 리포트 재조회 최소화**
  - 루틴 로그 변경 시 invalidate 대신 **캐시 직접 갱신**
  - 동일 데이터 스냅샷에는 반응하지 않도록 시그니처 기반 변경 감지

### 정량 결과(혼합 전략 적용 후)
- `routineReport/fetchByWeek` 호출:
  - **주간 이동 시에만 발생** (스냅샷 반복에도 추가 호출 없음)
- 루틴 로그 스냅샷:
  - 데이터 변화가 있을 때만 체크 반영
- 태스크 로그 구독:
  - 날짜 이동마다 구독 생성/해제 1회 (정상)

### 결론
- **실시간 동기화 요구사항을 만족하면서도**, 불필요한 리포트 재조회는 제거됨
- 캐시 기반 조회와 구독 기반 동기화를 분리하여 **성능과 기능 균형**을 확보

## 9. 기술 면접/포트폴리오용 핵심 요약

### 핵심 문제
- 홈 화면에서 월간 데이터가 실시간 구독으로 반복 생성/해제되어 비용과 트래픽이 누적됨
- 동일 날짜 재방문 시에도 네트워크 요청이 반복됨

### 해결 전략
- TanStack Query 도입으로 queryKey 설계 및 캐시 정책(staleTime, gcTime) 수립
- 월간 구독 → 단발 fetch 전환
- 날짜별 캐시 hit 확보(`staleTime=2분`, `refetchOnMount=false`)
- 측정 로그 기반으로 전/후 성능 지표 비교 체계 마련

### 구체적인 개선 방법
- **queryKey 규칙 정의**: `tasks/userId/date`, `taskLogs/userId/date`, `tasks/userId/month` 형태로 일관화해 캐시 재사용이 가능하도록 설계\n
- **구독 → fetch 전환**: 월간 캘린더 데이터는 실시간 구독을 제거하고 `get*ByMonthOnce` 단발 조회 함수로 교체\n
- **캐시 정책 튜닝**: 날짜별 조회는 `staleTime=2분`, `refetchOnMount=false`로 재방문 시 네트워크 요청 방지\n
- **옵티미스틱 업데이트 적용**: 태스크 생성/정렬/로그 토글 시 query cache를 먼저 업데이트해 즉시 반영\n
- **정량 계측 코드 삽입**: `perfBaseline` 로거로 fetch/subscribe 횟수, 소요 시간, 캐시 상태를 기록

### 정량 성과
- 월간 구독 subscribe: **1~5회 → 0회 (100% 감소)**
- 동일 날짜 재방문 fetch: **발생 → 0회 (100% 감소)**
- 루틴 주간 리포트 응답 시간: **248~403ms → 117~326ms (최대 71% 개선)**

### 한 줄 요약(포트폴리오)
- “실시간 구독 기반 패칭을 Query 캐싱 구조로 재설계해 월간 구독을 제거하고, 동일 날짜 재방문 시 fetch를 100% 줄였으며, 루틴 리포트 응답을 최대 71% 개선했다.”
